KEY FINDINGS
============

Finding 1: one-backend mcp_router does NOT add a prefix
  - With one server, tools appear as: tool1, tool2, tool3
  - With two backends, tools appear as: server1__tool1, server2__tool1, ...
  - The prefix is only added to disambiguate multiple servers

Finding 2: one-backend mcp_router is a transparent pass-through
  - Does not validate tool names
  - Any tools/call is forwarded to the one configured server with the full name intact
  - Calling server2__tool1 on a one-backend listener routes to server1 and returns
    "server1 ran server2__tool1" (no error)
  - Tool name isolation requires ext_authz (C2) or MCPVirtualServer

Practical implication:
  - Two-listener approach gives different tool naming conventions per user
    (alice: "tool1", bob: "server1__tool1" for the same underlying tool)
  - This naming inconsistency makes the two-listener approach unsuitable for production
    per-user filtering without a consistent naming contract
  - MCPVirtualServer (Kuadrant) solves this properly at the aggregation layer
